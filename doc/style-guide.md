# Style Guide

本ドキュメントは、スタイルの判断基準を定める。細部の仕様を網羅するものではない。

## 1. 基本方針

- **実用主義**: 読みやすさと公開の速さを最優先する。完璧を待たない。
- **最小限のコンポーネント**: ボタン・パネル・ヘッダー・フッター・サイドバーで原則すべてを構成する。新しいコンポーネントを作る前に、既存の組み合わせで済まないか考える。
- **無駄なし**: アニメーションや複雑なインタラクションは原則入れない。

## 2. 色の判断基準

Tailwind CSS の `sky` と `slate` を基本とする。
`src/styles/global.css` では `@theme` を使い、レイアウト・ダークモード対応用の CSS 変数（`--color-bg` など）を定義しているが、これらは Tailwind の `sky` / `slate` から導出される。

- **メインの強調**（リンク、ボタン、現在位置）: `sky` 系
- **背景・テキスト・枠線**（情報の下地と階層）: `slate` 系
- **アクセント**（タグ、ハイライト）: `sky` の薄い色を `slate` の濃いテキストと組み合わせる

色を選ぶ際は、上記2色軸に収まるかを最初に確認する。

## 3. タイポグラフィの判断基準

- **フォント**: システムフォントスタックのみ。Webフォントの読み込みは行わない。
  - **例外**: OG画像生成時のみ、Google Fonts（`Noto Sans JP`）を使用して日本語の正確なレンダリングを行う。
- **階層**: 情報の優先度を、サイズと字重で表現する。極端なサイズの乱用は避ける。
- **行間**: 本文は読みやすい程度に広めに取る。

## 4. レイアウトの判断基準

- **全ページ共通レイアウト**: ヘッダー・フッター・サイドバーは全ページで共通とする。
- **最大幅**: ありすぎずなさすぎず。極端に広げない。
- **余白**: 要素間の距離で情報のグルーピングを示す。詰めすぎず、離しすぎず。
- **モバイルファースト**: まずモバイルを想定し、必要に応じて広い画面用の調整を加える。

## 5. CSSの定義場所と判断基準

スタイルの定義場所は、影響範囲と再利用性で決める。

1. **`@theme`（デザイントークン層）**
   - `src/styles/global.css` の `@theme` ブロック内に配置する。
   - 色、フォント、カスタム変数（`--color-bg` など）のみを定義する。
   - 色の判断基準（`sky` / `slate`）をここに反映させる。

2. **`@layer base`（要素デフォルト層）**
   - `src/styles/global.css` の `@layer base` ブロック内に配置する。
   - `body`、`a` など、HTML要素の最小限のリセット・デフォルトのみを定義する。
   - 特定のコンポーネントやページに依存するスタイルは入れない。

3. **`@layer components`（再利用パターン層）**
   - `src/styles/global.css` または独立したCSSファイル（例：`prose.css`）に配置する。
   - 複数のコンポーネント・ページで共通に使われる**抽象パターン**のみを定義する。例：`.card`、`.badge`、`.prose`。
   - ページ固有・用途固有のスタイルはここに入れない。例：スライド専用の `.slide-*` は除外する。
   - 命名規則：パターン名は具体名（`.card`）、状態は修飾子（`.interactive-card`）で表現する。

4. **Astroコンポーネント / ページのクラス属性（レイアウト・配置層）**
   - 構造（`flex`, `grid`）、余白（`m-`, `p-`）、位置（`justify-`, `items-`）、サイズ（`w-`, `h-`）などは、Tailwindのユーティリティクラスを使う。
   - `style` 属性は**原則禁止**。ただし、OG画像の固定サイズ（`width: 1200px`）など、計算不可能な絶対値や外部フォント指定には例外を設ける。
   - 同じユーティリティの組み合わせが3箇所以上で繰り返される場合は、`@layer components` にパターンとして抽出する。

## 6. やらないこと

以下は v0.1 では行わない。

- アニメーションやトランジション（ただし、hover での色変化など最低限のフィードバックは許容する）
- 完璧なデザインシステムやコンポーネントライブラリ化
- 重いインタラクティブ機能
- コンポーネントファイルに見た目の詳細を閉じ込める（`.card` や `.badge` のような共通パターンは `@layer components` に置き、一貫性を保つ）

## 7. 迷ったときの基準

スタイルの判断に迷った場合、以下を優先順に確認する。

1. **本書の「基本方針」に反していないか**
2. **既存のコンポーネントクラス（`.card`、`.badge`、`.prose` など）で実現できないか**
3. **sky と slate の範囲で収まるか**
4. **モバイルで見たときに支障がないか**
5. **`style` 属性を使う必要があるか。Tailwindのユーティリティクラスまたはコンポーネントクラスで代用できないか**

上記を満たすなら、細部の揺らぎは許容して先に進む。完璧を待たない。
