# User Story Mapping

## バージョン定義

| バージョン | コードネーム | 目的 |
| - | - | - |
| **v0.1** | MVP | 「書いてデプロイする」最小パイプラインの確立。中間成果物を放置せず、小さな成果として公開する土台を構築する。 |
| **v0.2** | Refine | CI/CD の自動化と、既存コンテンツ構造の再設計。イベント参加記録の表示方法を改善し、記事の series 機能をストーリーに統合して重複を解消する。 |
| **v0.3** | Narrative | コンテンツの多様化と高付加価値化。単発の記事に加え、流れを持つストーリーや構造化されたスライドを扱えるようにし、蓄積した知見の再構成・再活用を可能にする。 |

## ユーザーストーリーマップ

### 1. 執筆する（Write）

中間成果物から知見を抽出し、コンテンツを執筆する。

- [x] **技術記事を書く**
  - 中間成果物から知見を抽出し、記事として執筆・公開したい。なぜなら、検証結果や試行記録を「物語として完成させ」、小さな成果として資産化したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 記事を所定の形式で記述すれば、自動的に公開されること。
    2. `Version: v0.1` 指定されたスタイルガイドに従った装飾が適用され、可読性のあるレイアウトとなっていること。

- [x] **イベント参加記録を書く**
  - 参加したイベントの履歴や簡単な説明を記録したい。なぜなら、外部との接点や学びを軽く残し、自分の活動の軌跡を可視化したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` イベント参加記録を所定の形式で記述すれば、自動的に公開されること。
    2. `Version: v0.1` イベント名・開催日・参加概要などの基本情報が一覧で確認できること。個別の詳細画面は持たない。
    3. `Version: v0.2` 同じイベントへの複数回参加を、1つの記録にまとめられること。
    4. `Version: v0.2` イベント参加記録は、技術記事やスライドなどとは別の仕組みで管理・表示されること。

- [ ] **スライド形式で情報を構造化して公開する**
  - 中間成果物から知見を抽出し、スライド形式で構造化して公開したい。なぜなら、同じフォーマットで多様な表現ができ、自分のペースで蓄積していきたいからだ。
  - **受け入れ条件**
    1. `Version: v0.3` スライドを所定の形式で記述すれば、自動的にスライド形式で公開されること。
    2. `Version: v0.3` ページ送りやスライド特有のレイアウトが機能すること。
    3. `Version: v0.3` 詳細ページは、概要パネルとスライド本体で構成されること。

- [ ] **ストーリーを書く**
  - 一連の流れのあるコンテンツ群とその背景情報をまとめて公開したい。なぜなら、断片的な知見を連続性のある形で再構成し、より深い文脈を伝えたいからだ。
  - **受け入れ条件**
    1. `Version: v0.3` ストーリーを所定の形式で記述すれば、自動的に公開されること。
    2. `Version: v0.3` 背景情報と、流れのあるコンテンツ群が一覧・詳細両方で確認できること。
    3. `Version: v0.3` 詳細ページは、概要パネルとストーリー本文で構成されること。

- [x] **シンプルな装飾を適用する**
  - 過度に完璧でないが、読みやすい装飾を施したい。なぜなら、完璧を狙いすぎて公開を止めることを避けつつ、最低限の品質は担保したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 指定されたスタイルガイドに従い、適切な装飾が適用されること。
    2. `Version: v0.1` 適用後の表示が、モバイル・デスクトップ両方で可読性を保つこと。

- [x] **全コンテンツを一覧で見る**
  - 技術記事・スライド・ストーリーなど、すべての主要コンテンツを一覧で確認したい。なぜなら、自分のアウトプット全体を時系列や種別を横断して俯瞰したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` すべての主要コンテンツが時系列で混在して一覧表示されること。
    2. `Version: v0.1` 一覧上でコンテンツ種別（技術記事・スライド・ストーリー）が視覚的に区別できること。

- [x] **概要パネルを使う**
  - 各コンテンツのメタデータを共通のパネルで表示したい。なぜなら、どの画面からでもコンテンツの概要を同じ形式で把握でき、認知負荷を下げたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 一覧の各アイテムに、概要パネルが表示されること。
    2. `Version: v0.1` 各詳細ページ（技術記事・スライド・ストーリー）のトップに、概要パネルが表示されること。
    3. `Version: v0.1` 概要パネルには、タイトル・説明・公開日・更新日・タグ・カバー画像・コンテンツ種別バッジが表示されること。

- [x] **即座に記録する**
  - 学びが揮発する前に、最小限の手順でドラフトを作成したい。なぜなら、中間成果物の育成を停止せず、開発サイクルを回し続けたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` テンプレートまたは雛形から、数ステップで新規ドラフトを作成できること。
    2. `Version: v0.1` ドラフトは即座にプレビュー環境で確認できること。

- [x] **テンプレートから新規ドラフトを作成する**
  - 執筆を開始する際、テンプレートをコピーするだけで定型の雛形を用意したい。なぜなら、書き始めるまでの認知負荷と品質への過度な意識を下げ、すぐに中身を書き始めたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 各コンテンツタイプのテンプレートが用意されていること。
    2. `Version: v0.1` テンプレートには、必須情報と本文の最低限の構造が含まれていること。

### 2. 公開する（Publish）

執筆したコンテンツを迅速にビルドし、インターネット上に公開する。

- [x] **サイトをビルドする**
  - 執筆したコンテンツを公開用の形に変換したい。なぜなら、高速・安価・シンプルに公開するためだ。
  - **受け入れ条件**
    1. `Version: v0.1` ビルドがエラーなく完了すること。
    2. `Version: v0.1` ビルド結果として、公開に必要なすべてのファイルが生成されること。

- [x] **インターネット上にデプロイする**
  - ビルド成果物をインターネット上に公開し、誰からでもアクセスできるようにしたい。
  - **受け入れ条件**
    1. `Version: v0.1` ビルド成果物が公開用の場所にアップロードされること。
    2. `Version: v0.1` アップロード後、公開URLから実際にアクセスできること。

- [x] **最小手順でデプロイする**
  - 公開のハードルを下げるため、最小ステップの手順でデプロイしたい。なぜなら、完璧を待たずに小さく早く公開することを最重視したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 手動操作を含めた最小ステップのデプロイ手順が文書化されていること。
    2. `Version: v0.1` 文書化された手順に従い、実際に新規コンテンツを公開できること（デプロイの自動化は v0.1 のスコープ外とし、手動デプロイを許容する）。

- [x] **自動でデプロイする**
  - 変更を加えた際、手動操作なしで公開まで完結するようにしたい。なぜなら、手間を減らし、公開のハードルをさらに下げ、継続的なリリースを可能にするためだ。
  - **受け入れ条件**
    1. `Version: v0.2` ソースコードの変更がリポジトリに反映された際、自動的に公開されること。
    2. `Version: v0.2` コンテンツの変更がリポジトリに反映された際、自動的に公開されること。
    3. `Version: v0.2` ソースコードの変更でバージョンが更新されること。

- [x] **基本的な SEO 対応を行う**
  - 完璧を狙いすぎない範囲で、検索エンジンに認識される基本的なメタ情報を設定したい。なぜなら、価値提供のため最低限の可視性は確保したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 各ページに、タイトル・説明文・OGP画像等の基本的なメタ情報を設定できること。
    2. `Version: v0.1` 設定されたメタ情報が、検索エンジンに正しく認識される状態になっていること。

- [x] **OGP 画像を自動生成する**
  - OGP 画像を指定しなくても、ビルド時に自動で OGP 画像が生成されるようにしたい。なぜなら、画像を毎回手作りするコストが公開のハードルを上げ、中間成果物の回収を妨げるからだ。
  - **受け入れ条件**
    1. `Version: v0.1`（v0.2 から前倒しで実装済み）OGP 画像を手作りしなくても、自動で生成されること。
    2. 自動生成される OGP 画像には、タイトル・公開日・サイト名・タグなどの基本情報が含まれること。

- [ ] **スライドを PDF として出力する**
  - 構造化されたスライドコンテンツを、印刷・共有用の PDF としてダウンロードできるようにしたい。なぜなら、同じフォーマットで蓄積した知見を別の場面でも再利用したいからだ。
  - **受け入れ条件**
    1. `Version: v0.3` スライドコンテンツを PDF としてダウンロードできること。

### 3. 振り返る・還流させる（Reflect & Circulate）

公開されたコンテンツを通じてフィードバックを得（得なくてもよい）、次の中間成果物の育成に還流させる。

- [ ] **アクセス解析を確認する**
  - 公開コンテンツの閲覧トレンドを簡易的に把握したい。なぜなら、フィードバックの有無にかかわらず、蓄積と過去の自己との比較に判断基準を切り替え、成長実感を得たいからだ。
  - **受け入れ条件**
    1. `Version: v0.3` アクセス解析が確認できること。

- [x] **SNS へ共有する**
  - 公開したコンテンツを外部に伝達したい。なぜなら、小さな成果の蓄積自体が資産であり、次の挑戦への還流のきっかけとなるからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 各ページから、指定されたSNSへ共有できること。

- [x] **中間成果物からの抽出を続ける**
  - 公開後も「未完成でも公開する」「一定のリズムで形にする」というプロセスを継続したい。なぜなら、循環し続けること自体が成長であるからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 公開後も「未完成でも公開する」「一定のリズムで形にする」というプロセスが継続できる環境であること。
    2. `Version: v0.1` 中間成果物からコンテンツを執筆し、公開するまでの一連の手順が文書化されていること。

## 制約・やらないこと

以下は Inception Deck に基づき、プロジェクト全体のスコープから除外する。

- CMS 的な編集・管理機能は提供しない（書いてビルドして公開するだけ）。
- 他者向けのコメント機能や SNS 連携機能（シェアボタンを除く）は提供しない。
- 重いアニメーションや複雑なインタラクティブ機能は実装しない。
- 完璧なデザインシステムは構築しない。
- SEO 最適化やアクセス解析を完璧に仕上げることは目指さない。ただし、価値提供のため基本的な SEO 対応は行う。完璧を狙いすぎて公開を止めることだけはやらない。
- 美しい Keynote レベルの演出を持つ Web スライドは目指さない。
- 汎用的なツール化は行わず、自分専用のプラットフォームとして運用する。

## 参考：採用する技術スタック（受け入れ条件の対象外）

本セクションは参考情報であり、受け入れ条件の検証対象とはしない。

- **フレームワーク**: Astro
- **コンテンツフォーマット**: MDX
- **スタイリング**: Tailwind CSS
- **ホスティング・CDN**: Cloudflare R2 + CDN
- **アクセス解析**: Cloudflare Web Analytics
- **SNS連携**: X（旧 Twitter）へのシェアボタン
- **OGP 自動生成・スライド PDF 化**: Playwright（ビルド時処理）

上記技術の採用は Inception Deck で決定されたものであるが、受け入れ条件は「何ができているか」に焦点し、特定の技術名に依存しない記述とする。
