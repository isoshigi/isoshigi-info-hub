# User Story Mapping

## バージョン定義

| バージョン | コードネーム | 目的 |
| - | - | - |
| **v0.1** | MVP | 「書いてデプロイする」最小パイプラインの確立。中間成果物を放置せず、小さな成果として公開する土台を構築する。 |
| **v0.2** | Enhance | コンテンツの多様化と高付加価値化。単発の記事に加え、流れを持つストーリーや構造化されたスライドを扱えるようにし、蓄積した知見の再構成・再活用を可能にする。 |

## ユーザーストーリーマップ

### 1. 執筆する（Write）

中間成果物から知見を抽出し、コンテンツを執筆する。

- [ ] **技術記事を書く**
  - 中間成果物から知見を抽出し、記事として執筆・公開したい。なぜなら、検証結果や試行記録を「物語として完成させ」、小さな成果として資産化したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 所定のディレクトリにファイルを配置するだけで、技術記事として正しくレンダリングされること。
    2. `Version: v0.1` 指定されたスタイルガイドに従った装飾が適用され、可読性のあるレイアウトとなっていること。

- [ ] **イベント参加記録を書く**
  - 参加したイベントの履歴や簡単な説明を記録したい。なぜなら、外部との接点や学びを軽く残し、自分の活動の軌跡を可視化したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 所定のディレクトリにファイルを配置するだけで、イベント参加記録として正しくレンダリングされること。
    2. `Version: v0.1` イベント名・開催日・参加概要などの基本情報が一覧で確認できること。個別の詳細画面は持たず、共通一覧に表示するのみとする。

- [ ] **スライド形式で情報を構造化して公開する**
  - 中間成果物から知見を抽出し、スライド形式で構造化して公開したい。なぜなら、同じフォーマットで多様な表現ができ、自分のペースで蓄積していきたいからだ。
  - **受け入れ条件**
    1. `Version: v0.2` 所定のディレクトリにファイルを配置するだけで、スライド形式として正しくレンダリングされること。
    2. `Version: v0.2` ページ送りやスライド特有のレイアウトが機能すること。
    3. `Version: v0.2` 詳細ページは、概要パネルとスライド本体で構成されること。

- [ ] **ストーリーを書く**
  - 一連の流れのあるコンテンツ群とその背景情報をまとめて公開したい。なぜなら、断片的な知見を連続性のある形で再構成し、より深い文脈を伝えたいからだ。
  - **受け入れ条件**
    1. `Version: v0.2` 所定のディレクトリにファイルを配置するだけで、ストーリーとして正しくレンダリングされること。
    2. `Version: v0.2` 背景情報と、流れのあるコンテンツ群が一覧・詳細両方で確認できること。
    3. `Version: v0.2` 詳細ページは、概要パネルとストーリー本文で構成されること。

- [ ] **シンプルな装飾を適用する**
  - 過度に完璧でないが、読みやすい装飾を施したい。なぜなら、完璧を狙いすぎて公開を止めることを避けつつ、最低限の品質は担保したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 指定されたスタイルガイドに従い、ユーティリティクラスやスタイル定義が適用できること。
    2. `Version: v0.1` 適用後の表示が、モバイル・デスクトップ両方で可読性を保つこと。

- [ ] **全コンテンツを一覧で見る**
  - 技術記事・イベント参加記録・スライド・ストーリーなど、すべてのコンテンツを共通の一覧で確認したい。なぜなら、自分のアウトプット全体を時系列や種別を横断して俯瞰したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` `/pages` に、すべてのコンテンツが時系列で混在して一覧表示されること。
    2. `Version: v0.1` 一覧上でコンテンツ種別（技術記事・イベント・スライド・ストーリー）が視覚的に区別できること。

- [ ] **概要パネルを使う**
  - 各コンテンツのメタデータを共通のパネルで表示したい。なぜなら、どの画面からでもコンテンツの概要を同じ形式で把握でき、認知負荷を下げたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 共通一覧の各アイテムに、概要パネルが表示されること。
    2. `Version: v0.1` 各詳細ページ（技術記事・スライド・ストーリー）のトップに、概要パネルが表示されること。
    3. `Version: v0.1` 概要パネルには、タイトル・説明・公開日・更新日・タグ・カバー画像・コンテンツ種別バッジが表示されること。

- [ ] **即座に記録する**
  - 学びが揮発する前に、最小限の手順でドラフトを作成したい。なぜなら、中間成果物の育成を停止せず、開発サイクルを回し続けたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` テンプレートまたは雛形から、数ステップで新規ドラフトを作成できること。
    2. `Version: v0.1` ドラフトは即座にプレビュー環境で確認できること。

- [ ] **テンプレートから新規ドラフトを作成する**
  - 執筆を開始する際、テンプレートファイルをコピーするだけで定型のフロントマターと雛形を用意したい。なぜなら、書き始めるまでの認知負荷と品質への過度な意識を下げ、すぐに中身を書き始めたいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 所定のテンプレートディレクトリに、各コンテンツタイプの MDX テンプレートファイルが配置されていること。
    2. `Version: v0.1` テンプレートファイルには、必須フィールドを含むフロントマター雛形と、本文の最低限の構造が含まれていること。

### 2. 公開する（Publish）

執筆したコンテンツを迅速にビルドし、インターネット上に公開する。

- [ ] **静的サイトをビルドする**
  - 執筆したコンテンツを公開用の静的ファイルに変換したい。なぜなら、高速・安価・シンプルに公開するためだ。
  - **受け入れ条件**
    1. `Version: v0.1` ビルドコマンドを実行した際、エラーなく完了すること。
    2. `Version: v0.1` ビルド結果として、公開に必要なすべての静的ファイルが出力ディレクトリに生成されること。

- [ ] **公開用ストレージにデプロイする**
  - ビルド成果物を公開用ストレージにアップロードし、CDN 経由で高速に配信したい。
  - **受け入れ条件**
    1. `Version: v0.1` ビルド成果物が指定された公開用ストレージにアップロードされること。
    2. `Version: v0.1` アップロード後、公開URLから実際にアクセスできること。

- [ ] **最小手順でデプロイする**
  - 公開のハードルを下げるため、最小ステップの手順でデプロイしたい。なぜなら、完璧を待たずに小さく早く公開することを最重視したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 手動操作を含めた最小ステップのデプロイ手順が文書化されていること。
    2. `Version: v0.1` 文書化された手順に従い、実際に新規コンテンツを公開できること（デプロイの自動化は v0.1 のスコープ外とし、手動デプロイを許容する）。

- [ ] **基本的な SEO 対応を行う**
  - 完璧を狙いすぎない範囲で、検索エンジンに認識される基本的なメタ情報を設定したい。なぜなら、価値提供のため最低限の可視性は確保したいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 各ページに、タイトル・説明文・OGP画像等の基本的なメタ情報を設定できること。
    2. `Version: v0.1` 設定されたメタ情報が、HTMLソース上に正しく出力されていること。

- [ ] **OGP 画像を自動生成する**
  - OGP 画像を指定しなくても、ビルド時に自動で OGP 画像が生成されるようにしたい。なぜなら、画像を毎回手作りするコストが公開のハードルを上げ、中間成果物の回収を妨げるからだ。
  - **受け入れ条件**
    1. `Version: v0.1`（v0.2 から前倒しで実装済み）ビルド時に Playwright を用いて各ページ用の OGP 画像（1200×630px）が自動生成され、`/img/.../og.png` に出力されること。
    2. 自動生成される OGP 画像には、タイトル・公開日・サイト名・タグなどの基本情報が含まれること。

- [ ] **スライドを PDF として出力する**
  - 構造化されたスライドコンテンツを、印刷・共有用の PDF としてダウンロードできるようにしたい。なぜなら、同じフォーマットで蓄積した知見を別の場面でも再利用したいからだ。
  - **受け入れ条件**
    1. `Version: v0.2` ビルド時に、スライドコンテンツから PDF が生成されること。
    2. `Version: v0.2` スライド詳細ページに、生成された PDF のダウンロードリンクが配置されていること。

### 3. 振り返る・還流させる（Reflect & Circulate）

公開されたコンテンツを通じてフィードバックを得（得なくてもよい）、次の中間成果物の育成に還流させる。

- [ ] **軽量なアクセス解析を確認する**
  - 公開コンテンツの閲覧トレンドを簡易的に把握したい。なぜなら、フィードバックの有無にかかわらず、蓄積と過去の自己との比較に判断基準を切り替え、成長実感を得たいからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 軽量なアクセス解析用のスクリプトが、全ページに埋め込まれていること。
    2. `Version: v0.1` ダッシュボードで、ページ別の閲覧トレンドが確認できること。

- [ ] **SNS へ共有する**
  - 公開したコンテンツを外部に伝達したい。なぜなら、小さな成果の蓄積自体が資産であり、次の挑戦への還流のきっかけとなるからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 全ページに、指定されたSNSへのシェアボタンが配置されていること。
    2. `Version: v0.1` シェアボタンを押下した際、該当ページのタイトルとURLが含まれた状態でSNSの投稿画面が開くこと。

- [ ] **中間成果物からの抽出を続ける**
  - 公開後も「未完成でも公開する」「一定のリズムで形にする」というプロセスを継続したい。なぜなら、循環し続けること自体が成長であるからだ。
  - **受け入れ条件**
    1. `Version: v0.1` 公開後も「未完成でも公開する」「一定のリズムで形にする」というプロセスが継続できる環境であること。
    2. `Version: v0.1` 中間成果物からコンテンツを執筆し、ビルド・デプロイするまでの一連の手順が、リポジトリ内または所定のドキュメントディレクトリに文書化されていること。

## 制約・やらないこと（v0.1 時点）

以下は Inception Deck に基づき、v0.1 のスコープから除外する。

- CMS 的な編集・管理機能は提供しない（書いてビルドして公開するだけ）。
- 他者向けのコメント機能や SNS 連携機能（シェアボタンを除く）は提供しない。
- 重いアニメーションや複雑なインタラクティブ機能は実装しない。
- 完璧なデザインシステムは構築しない。
- SEO 最適化やアクセス解析を完璧に仕上げることは目指さない。ただし、価値提供のため基本的な SEO 対応は行う。完璧を狙いすぎて公開を止めることだけはやらない。
- 美しい Keynote レベルの演出を持つ Web スライドは目指さない。
- 汎用的なツール化は行わず、自分専用のプラットフォームとして運用する。

## 参考：採用する技術スタック（受け入れ条件の対象外）

本セクションは参考情報であり、受け入れ条件の検証対象とはしない。

- **フレームワーク**: Astro
- **コンテンツフォーマット**: MDX
- **スタイリング**: Tailwind CSS
- **ホスティング・CDN**: Cloudflare R2 + CDN
- **アクセス解析**: Cloudflare Web Analytics
- **SNS連携**: X（旧 Twitter）へのシェアボタン
- **OGP 自動生成・スライド PDF 化**: Playwright + html-to-image（ビルド時処理）

上記技術の採用は Inception Deck で決定されたものであるが、受け入れ条件は「何ができているか」に焦点し、特定の技術名に依存しない記述とする。
